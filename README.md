# of-codex-agent

A minimal **OpenAI Codex CLI** demo whose model backend is served by
**Azure AI Foundry**.

It is the Codex sibling of [`of-ghcp-agent`](../of-ghcp-agent) (GitHub Copilot)
and [`of-pi-agent`](../of-pi-agent) (Pi). Like `of-ghcp-agent`, it is stripped
down to **keep only a skill** — but the integration target is different (see
[Why this differs from of-ghcp-agent](#why-this-differs-from-of-ghcp-agent)).

## What's here

```
.codex/skills/
└── joke/
    └── SKILL.md        ← tells Codex to respond like a pirate
AGENTS.md               ← project instructions Codex loads automatically
config.example.toml     ← Azure AI Foundry model-provider config (copy into ~/.codex/config.toml)
```

Codex auto-loads any subdirectory under `.codex/skills/` that contains a
`SKILL.md`, and reads `AGENTS.md` from the repo root. Nothing else from a full
sample is kept (no `Dockerfile`, `main.py`, `requirements.txt`, etc.).

## How Codex connects to Azure AI Foundry

Codex talks to **any OpenAI-compatible endpoint** via a custom `model_provider`
in `~/.codex/config.toml`. Azure AI Foundry exposes exactly such an endpoint, so
"connecting" Codex to Foundry is purely a config + auth step — no code.

### 1. Deploy a model in Azure AI Foundry

In the Foundry portal, deploy a model (e.g. `gpt-5`, `gpt-4o`) and note:

- **Deployment name** — this becomes Codex's `model`.
- **Endpoint** — `https://<resource>.openai.azure.com/openai` (or the unified
  `https://<resource>.services.ai.azure.com/openai`).
- **API key** and a supported **api-version**.

### 2. Configure Codex

Copy the relevant block from [`config.example.toml`](./config.example.toml) into
`~/.codex/config.toml`:

```toml
model_provider = "azure"
model = "gpt-5"                    # your Foundry *deployment name*

[model_providers.azure]
name = "Azure AI Foundry"
base_url = "https://<your-resource>.openai.azure.com/openai"
env_key = "FOUNDRY_API_KEY"
query_params = { api-version = "2025-04-01-preview" }
wire_api = "responses"             # "chat" for gpt-4o/gpt-4.1 deployments
```

Key points:

- `model` must be the **deployment name**; Codex appends it to `base_url` as
  `/deployments/<model>/...` automatically.
- `wire_api = "responses"` for GPT-5 / reasoning-class deployments;
  `"chat"` for chat-completions deployments (gpt-4o, gpt-4.1, …).
- Bump `api-version` if you hit a 404 / unsupported-version error.
- `env_key` is the **name of an environment variable**, not the key itself.
  Putting the raw key here triggers `Missing environment variable: '<key>'` —
  keep the name and `export` the key separately (step 3).

### 3. Provide the key and run

```bash
export FOUNDRY_API_KEY="<your-azure-ai-foundry-key>"

cd of-codex-agent
codex                              # interactive; or: codex exec "Tell me a joke"
```

Because the `joke` skill is auto-loaded, ask:

> Tell me a joke

…and the reply comes back in a swashbuckling pirate tone — generated by your
Azure AI Foundry deployment.

Sanity-check the wiring without editing your config:

```bash
codex doctor
codex exec -c model_provider=azure -c model=gpt-5 "ping"
```

## Why this differs from of-ghcp-agent

`of-ghcp-agent` **deploys** a GitHub Copilot agent *into* Azure AI Foundry
Hosted Agents (via [`open-foundry`](https://github.com/1openwindow/open-foundry)'s
`copilot-sdk` runtime). `open-foundry` currently ships runtimes for **Copilot**
and **Pi** only — there is **no Codex hosted-agent runtime** — so that
deploy-to-Foundry path is not available for Codex today.

This repo therefore demonstrates the inverse, equally common integration:

| | of-ghcp-agent | of-codex-agent |
| --- | --- | --- |
| Agent's role | runtime **deployed into** Foundry | **local client** of Foundry |
| Foundry's role | hosts/runs the agent | serves the **model** |
| Wiring | `open-foundry` deploy skill | `~/.codex/config.toml` provider |

In short: **of-ghcp-agent runs an agent *on* Foundry; of-codex-agent runs Codex
*against* a Foundry model.**

## Keeping the sample fresh

Your real provider config and secrets stay out of git: `config.toml`, `.env*`,
and editor/OS cruft are git-ignored. The tracked sample is only the skill,
`AGENTS.md`, `config.example.toml`, this README, and `.gitignore`.

```bash
git clean -fdX        # remove all ignored files
git status            # should report nothing to commit
```
