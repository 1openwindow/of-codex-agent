# of-codex-agent

A minimal **OpenAI Codex on Foundry hosted agent** demo. You bring this repo;
[`open-foundry`](https://github.com/1openwindow/open-foundry) deploys it into
**Microsoft Foundry Hosted Agents** on the `codex-foundry-runtime` image.

It exists to exercise the **OpenAI Codex harness** end-to-end on Foundry — the
runtime drives [`@openai/codex-sdk`](https://www.npmjs.com/package/@openai/codex-sdk)
in-process, maps each Foundry invocation onto a Codex thread, and streams the
reply back. The repo is stripped down to **keep only a skill** (plus the
`AGENTS.md` Codex loads automatically).

## What's here

```
.codex/skills/
└── joke/
    └── SKILL.md        ← tells Codex to respond like a pirate
AGENTS.md               ← project instructions Codex loads automatically
config.example.toml     ← Azure AI Foundry model-provider config for running Codex locally
```

The Codex harness auto-loads any subdirectory under `.codex/skills/` that
contains a `SKILL.md`, and reads `AGENTS.md` from the repo root. Nothing else
from a full sample is kept (no `main.py`, `requirements.txt`, etc.). The
`Dockerfile`, `azure.yaml`, `agent.yaml`, `agent.manifest.yaml`, and
`.dockerignore` that `open-foundry` generates on deploy are git-ignored — this
repo tracks only the sample.

## Deploy to Foundry with open-foundry

You bring this repo; the [`open-foundry`](https://github.com/1openwindow/open-foundry)
skill does the rest. Run it from the **Codex CLI** — this sample's own harness —
so the deploy is exercised end-to-end:

```bash
git clone https://github.com/1openwindow/of-codex-agent
cd of-codex-agent
npx skills add 1openwindow/open-foundry --agent codex -y   # install the deploy skill (once)
codex                                      # open the Codex CLI here
```

In the session, say:

> Deploy this Codex agent to Foundry.

The skill confirms the agent name and the `codex-foundry-runtime` image, then
bootstraps the standard `azd` files and runs `azd deploy`. Codex BYOK is
**API-key only** — provide a Foundry OpenAI-compatible endpoint, model, and key
(keyless managed identity is not supported on this harness).

Once it's running, invoke the agent from this repo:

```bash
azd ai agent invoke of-codex-agent '{"input": "Tell me a joke"}'
```

The reply comes back in a swashbuckling pirate tone — generated by your Azure AI
Foundry deployment, through Codex.

## Run Codex locally against a Foundry model

Same wiring, no deploy: point the local Codex CLI at your Foundry model. Copy the
relevant block from [`config.example.toml`](./config.example.toml) into
`~/.codex/config.toml`:

```toml
model_provider = "azure"
model = "gpt-5"                    # your Foundry *deployment name*

[model_providers.azure]
name = "Azure AI Foundry"
base_url = "https://<your-resource>.openai.azure.com/openai"
env_key = "FOUNDRY_API_KEY"
query_params = { api-version = "2025-04-01-preview" }
wire_api = "responses"             # "chat" for gpt-4o/gpt-4.1 deployments
```

`model` is the **deployment name** (Codex appends it as `/deployments/<model>/…`);
`env_key` is the **name** of an env var, not the key itself. Then:

```bash
export FOUNDRY_API_KEY="<your-azure-ai-foundry-key>"
cd of-codex-agent
codex                              # interactive; or: codex exec "Tell me a joke"
```

## Testing repeatedly (keep the sample fresh)

The files open-foundry generates on deploy (`Dockerfile`, `.dockerignore`,
`azure.yaml`, `agent.yaml`, `agent.manifest.yaml`) plus `.azure/` and your local
`config.toml` are **git-ignored** — so deploying never dirties the tracked sample
and you cannot accidentally commit them.

```bash
git clean -fdX             # remove all ignored files (including .azure/)
git clean -fdX -e .azure   # ...but keep your azd env for a faster re-deploy
```

Afterwards `git status` should report nothing to commit.
